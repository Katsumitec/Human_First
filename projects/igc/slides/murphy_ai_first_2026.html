Background

GC 第三方遊戲整合平台

AI 導入不是單純「寫 Code 快一點」，而是在複雜工程流程中重新分配人與 AI 的注意力。

05
Review Workflow

案例二：AI Review 從「問一次」升級為多層檢查

普通檢查

/codex:review

檢查常見錯誤、可讀性、結構問題。

攻擊式檢查

/codex:adversarial-review

刻意從找錯、挑戰、攻擊角度檢查。

特定二審

/specific-task-review

針對金流、狀態機、冪等性等高風險項目二審。

重點不是「AI Review 一次」，而是設計不同角色、不同角度的檢查流程。
06
Pitfall

踩坑一：grep 只看前幾行，導致 AI 誤判

發生什麼事

AI 用 grep 快速搜尋程式碼，但只看到片段與前幾行，沒有掌握完整函式、類別與引用關係。

結果

AI 的推論看起來合理，但其實建立在不完整上下文上。

解法

導入 Codegraph MCP

讓 AI 更容易理解程式碼結構、呼叫關係、類別引用與實際依賴。

這次學到：AI 很強，但它需要的是「有意義的上下文」，不是更多雜訊。
07
Pitfall

踩坑二：AI 很會猜，但它沒有全部資訊

例如 AWS S3 存取安全，AI 依照主流架構建議使用 OAC。

CloudFront + S3OAC主流建議
但公司現況不同

實際 CDN 使用 Cloudflare，並且基於成本與亂數檔名策略做安全性取捨。這不一定是漏洞，而是 Trade-off。

AI 的建議通常合理，
但不一定符合公司現況。
08
Pitfall

踩坑三：注意 AI 的知識截止時間

高風險資訊

工具版本、套件 API、雲端規格、價格、最佳實踐。

常見問題

AI 可能把過時資訊講得很有信心。

處理方式

涉及最新規格或決策時，需要查證官方文件或實測。

AI 可以加速研究，但不能取代驗證。
09
Pitfall

踩坑四：AI 會用 Mock 假裝完成核心機制

需求

接入單一錢包供應商，需要實作完整 Token 驗證機制。

AI 實作

為了讓流程跑通，直接使用 Mock / Stub，核心驗證其實沒有完成。

禁止 Fake Implementation： 除非任務明確要求，否則不得用 mock / stub / fake 取代正式邏輯。 Mock 只能存在於 test / local / sandbox，不得出現在 production path。
10
Principle

原理一：有意義的上下文，決定 AI 的關注點

Context 太少

AI 開始猜，輸出看似合理但可能偏離實際需求。

Context 太多

Attention 被稀釋，重要規則可能被淹沒。

AI 品質 = 模型能力 × Context 品質

Principle

原理二：CLAUDE.md、Skill 與任務規則要分層

Global Rules
CLAUDE.md
→
Domain Rules
Skill
→

本次限制
Global

團隊共同規範、架構原則、通用限制。

Domain

金流、遊戲整合、Review 等領域專屬注意事項。

Task

這次任務不得修改 API、不得變更 DB 等特殊限制。

12
Principle

原理三：固定流程要固化，不要每次重新問 AI

適合 Script / Tool

查訂單、查餘額、跑測試、產生固定結構、執行可重複流程。

適合 AI

架構討論、風險分析、文件理解、Trade-off、問題展開。

讓 AI 負責推理，讓 Script 負責執行，讓 Test 負責驗證，人類負責簽核。
13
Principle

原理四：AI 有 Mainstream Bias，但主流不一定最適合

主流做法

RD Done → Close

我們的流程

RD Done → 待驗收 → PM 驗收 → Close

不要問「主流是什麼」，先問「這條規則保護了什麼」。
14
Principle

原理五：LLM 是機率系統，不是確定性系統

Autopilot

Rule Engine + State Machine + Control Theory；同樣輸入，理論上得到穩定輸出。

LLM Agent

依機率產生下一個 Token；同樣問題可能得到不同解法。

高風險領域不該最大化 AI 自由度，而是最大化 AI 產出，同時最小化 AI 決策權。
15
Risk Control

金流與遊戲結算：AI Code Review 的重點是不變量

Money

餘額不得為負、不可重複扣款、金額不得使用 float。

State

狀態不可亂跳、不可逆轉、每次變更需可追蹤。

Idempotency

Callback、settle、rollback 必須可重複執行且結果一致。

不要只 Review Code，
要 Review 是否破壞資金與狀態的不變量。
16
Collaboration

PM 有全局，RD 只有局部：Spec 需要補上下文

常見問題

PM 腦中是完整遊戲大廳，但 RD 只看到單一 Spec，無法判斷切法是否合理。

改善方式

每張 Spec 必須包含 Epic、目的、關聯功能，讓 RD 能理解全貌與邊界。

AI 也一樣：沒有全貌，就只能根據片段猜測。
17
Decision

AI 不只產出內容，也能把技術資訊轉成決策資訊

工程師版

細節、Code、Log、SQL、架構推理。

主管版

問題、風險、成本、方案、優先級。

老闆版

做不做？花多少？多久？回報多少？

Markdown 只是內容；HTML / Dashboard / 顯示模板能讓決策者更快理解。

19
Conclusion

結論：AI First 不是讓 AI 做所有事

AI First 是不斷重新回答：
這件事到底應該由人思考，還是由 AI 思考？
AI

負責展開、生成、分析、找盲點。

System

負責規則、工具、流程、驗證。

Human

負責判斷、風險、責任、簽核。
