03
我們想要的協作模型

Component ↔ Figma Node，一個對一個

  • 換版不是「重畫一個畫面」，而是既有 Angular 元件 ↔ Figma 節點，一對一對應。
  • 前端原則（已寫進 /ui-to-code）：
    只動 .html / .scss，TS 邏輯幾乎不動——沿用 DOM、保留所有綁定。
  • 換版＝換「外觀＋間距＋比例」，不是換功能。
  • 對得上號 → 換版變成機械式、可預期、可批量的動作。

① 對得上 = 快

Node 對 class，改 scss 即生效

② 不動 TS = 安全

功能不會被換皮改壞

③ 可批量 = 產線化

多平台多皮膚邊際成本遞減

協作模型
04
給設計端

Figma 應該守的 5 條規範

  • ① 用 Auto Layout 重建組件（按鈕、卡片、列表項）
  • ② 用 Variables 抽出顏色和間距
  • ③ 用 Component + Variants 把重複元素（如 10 個彩票項）合併
  • ④ 用 Constraints 聲明元素在父容器的行為（拉伸／固定）
  • ⑤ 能做的話，再出 Mobile / Tablet 兩套畫布
這 5 條不是設計潔癖——每一條都對應前端一個工作步驟的成本。

全場核心

Figma 規範 ⇄ /ui-to-code 工作流

Figma 規範解決什麼對應 /ui-to-code沒做到的代價
① Auto Layout間距可被讀取Step 3 由 x/y/w/h、inset 推導 padding/gap逐節點手量座標
② Variables顏色/間距變數化Step 8 get_variable_defs → token、@include v()token 逆向猜，皮膚對不齊
③ Component+Variants重複元素一次定義節點→元件 1:1 9 宮格 / 10 彩票項複用 DOM重複圖層逐張比對易漏
④ Constraints父容器行為明確Step 3 比例 佔比/長寬比→%/aspect-ratio/flexRWD 變形、圖片拉伸
⑤ Mobile/TabletRWD 有設計依據web/app 雙變體 比例節奏對齊響應式靠前端臆測
設計多花 1 分鐘結構化，前端少花 1 小時逆向工程。

08
一條容易被誤解的規範

換膚機制：不是所有元件都要 token

  • 只有 lottery 元件才走「token＋皮膚 map＋@include v()」，要支援多皮膚。
  • 其餘元件（user-center、設定頁、活動頁…）無換膚需求 → scss 直接落 Figma 值。
  • 換色 ≠ 只換色值：色＋間距＋比例三者一起對齊；圖片/卡片維持長寬比避免變形。
  • krm 慣例：宇宙紫換版改 classic.scss 色票值，結構邊框硬編在共用層。

lottery 元件

token 化 → 一處改、多皮膚生效

非 lottery 元件

scss 直接落值 → 不過度抽象、改得快

落地成果

給外部設計的交付 Checklist

把這 5 條變成交稿驗收標準，前端收稿即可直接跑 /ui-to-code。

  • 卡片 / 按鈕 / 列表項都是 Auto Layout（非絕對定位）
  • 顏色、主要間距已抽成 Variables，命名清楚
  • 重複元素做成 Component + Variants，標明哪些是同一元件
  • 關鍵元素有 Constraints（拉伸 / 固定）
  • 提供節點層級連結（含 node-id，前端才能精準定位）
  • 重要頁面附 Mobile / Tablet 畫布
